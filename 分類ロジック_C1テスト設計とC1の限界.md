# 分類ロジックのホワイトボックステスト設計(C1網羅+C1の限界)

対象:DtcClassifier.Classify / ClassificationRule.Matches
評価項目5「カバレッジを考慮したテストケースを出せる(L2)」の立証材料。
L2の差別化点は「C1を満たせる」ことではなく、**「C1で測れるもの/測れないものを区別できる」**こと。後半の3パターンがその核。

---

## 1. テスト対象コード(分岐に番号を振る)

```csharp
public sealed class DtcClassifier
{
    private readonly IReadOnlyList<ClassificationRule> _rules;

    public DtcClassifier(IReadOnlyList<ClassificationRule> rules)
        => _rules = rules ?? throw new ArgumentNullException(nameof(rules));

    public FaultDomain Classify(DtcRecord record)
    {
        if (record is null)                                  // B1
            throw new ArgumentNullException(nameof(record));

        if (string.IsNullOrWhiteSpace(record.DtcCode))       // B2
            return FaultDomain.Unknown;

        foreach (var rule in _rules)                          // B3(ループ実行/0回)
        {
            if (rule.Matches(record))                         // B4
                return rule.Domain;                           // 先勝ち(first-match-wins)
        }
        return FaultDomain.Unknown;
    }
}

public sealed record ClassificationRule(string DtcCodePrefix, FaultDomain Domain)
{
    public bool Matches(DtcRecord record) =>
        record.DtcCode.StartsWith(DtcCodePrefix,              // B5
            StringComparison.OrdinalIgnoreCase);
}
```

C1(分岐網羅)の定義:**すべての判定について、真・偽の両方の出口を少なくとも1回ずつ通す**。
本コードの分岐出口一覧:B1(真/偽)、B2(真/偽)、B3(本体を1回以上実行/0回で抜ける)、B4(真/偽)、B5(真/偽)。

## 2. C1を満たすテストケース表

| TC | 入力(DtcCode / ルール集合) | 狙う分岐出口 | 期待値 |
|---|---|---|---|
| TC1 | record = null | **B1真** | ArgumentNullException |
| TC2 | DtcCode = ""(ルールは任意) | B1偽 → **B2真** | Unknown |
| TC3 | DtcCode = "   "(空白のみ) | **B2真**(IsNullOrWhiteSpaceの空白側) | Unknown |
| TC4 | "P0301" / ルール [("P0",Powertrain)] | B2偽 → B3実行 → **B4真・B5真**(先頭ルールで一致) | Powertrain |
| TC5 | "C1234" / ルール [("P0",Powertrain),("C",Chassis)] | **B4偽→真**(1本目不一致、2本目一致) | Chassis |
| TC6 | "U0100" / ルール [("P0",Powertrain)] | **B4偽・B5偽**のままループ終了 | Unknown |
| TC7 | "P0301" / ルール = 空リスト | **B3が0回**(ループ本体不実行) | Unknown |
| TC8 | コンストラクタに null | ガード節 真 | ArgumentNullException |

この8件で全分岐出口を網羅=C1 100%。補助として仕様確認用に:

| TC9 | "p0301"(小文字)/ ルール [("P0",Powertrain)] | 分岐は既出(B5真)だが**大文字小文字非依存の仕様**を固定 | Powertrain |

TC9はカバレッジを増やさない。「カバレッジに寄与しないが仕様を守るために必要なテスト」が存在すること自体、C1が十分条件でない証拠として面談で言及できる。

## 4. C1 100%でも見逃すバグ 3パターン

### パターン①:複合条件の組合せ漏れ(条件網羅の欠落)

「Active/Pendingのみ分類対象」という仕様でMatchesを拡張したとする。

```csharp
// 仕様:プレフィックス一致 かつ (Active または Pending) のとき一致
public bool Matches(DtcRecord r) =>
    r.DtcCode.StartsWith(DtcCodePrefix, StringComparison.OrdinalIgnoreCase)
    && (r.Status == DtcStatus.Active || r.Status != DtcStatus.Cleared);  // ★バグ:
    // 本来 r.Status == DtcStatus.Pending と書くべきところを != Cleared と誤記
```

次の2件でこの判定のC1は100%になる:
- T-a:プレフィックス一致 + Active → **真**の出口
- T-b:プレフィックス不一致 → **偽**の出口(短絡評価で右側は未評価)

しかし `Status = Unknown` のレコードは、正しい実装なら不一致、バグ版では一致——**この差が出る入力をどちらのテストも通っていない**。C1は「判定全体の真偽」しか見ないので、判定を構成する**個々の条件の組合せ**(ここではUnknownが右辺に到達するケース)は保証しない。
**検出するには**:条件ごとの真偽を独立に振るテスト(MC/DC的発想)。実務では「enumの全値を一巡させるTheory」が安価で効く。

### パターン②:境界値のオフバイワン

Activeが閾値**以上**のドメインに要注意フラグを立てる集計仕様で:

```csharp
// 仕様:activeCount が threshold 以上なら要注意
bool isCritical = summary.ActiveCount > threshold;   // ★バグ:>= と書くべきところを >
```

- T-a:ActiveCount=0, threshold=3 → false(**偽**の出口)
- T-b:ActiveCount=10, threshold=3 → true(**真**の出口)

C1は100%。だが `ActiveCount == 3` のとき、仕様はtrue・実装はfalseで**境界のちょうど上でだけ**間違う。C1は「両方の出口を通ったか」を見るだけで、**どの値で出口が切り替わるか**は見ない。
**検出するには**:境界値分析を併用し、境界そのもの(=3)と両隣(2, 4)を必ずテストに入れる。「カバレッジ網羅とテスト技法(境界値分析)は別の道具で、併用が前提」という語りに直結する。

### パターン③:先勝ちルールの順序依存(データ依存バグ)

```csharp
// ルール定義(コードではなく設定データ側のバグ)
var rules = new List<ClassificationRule>
{
    new("P0",  FaultDomain.Powertrain),
    new("P05", FaultDomain.Adas),        // ★到達不能:P05は常に上のP0が先に一致する
};
```

Classifyのコードは一切変えていないので、第2章のTC1〜TC8はそのまま全パスし、**C1は100%のまま**。しかし "P0562" はAdasに分類されるべきなのにPowertrainになる。カバレッジは**コードの分岐**を測る指標であり、**データ(ルール集合)の組合せ空間**は測定対象外——設定・ルールが振る舞いを決めるシステムでは、コードカバレッジ100%と正しさの距離が特に遠くなる。
**検出するには**:①本番ルールセットそのものを入力にした「ルール網羅テスト」(全ルールについて、そのルールでしか分類できない代表DTCを1つずつ流す)②「より特殊なプレフィックスが先に並ぶ」ことを検証するルール検証ロジックの追加(こちらは設計改善の話として語れる)。

## 5. 面談用まとめ(項目5をL2にする一言)

> 「C1 100%を達成しました(レポート提示)。ただしC1は『分岐の両出口を通ったか』しか保証しないので、**①複合条件の組合せ ②境界値の切り替わる位置 ③データ側の組合せ**は別途、条件網羅の発想・境界値分析・ルール網羅テストで補いました。カバレッジは品質の証明ではなく、**テストの抜けを見つけるための計器**として使っています。」
