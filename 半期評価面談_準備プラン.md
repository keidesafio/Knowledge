## 1. 15項目の分類と、実体験の紐づけ方

### a) 半期の業務実績から語れる項目

**項目4(コーディング L1→L2)← 業務①**
- 語り方:「前半はフォローを受けながら(=L1)だったが、最後の1か月は**自分で実装方針を立て、マージリクエストで指摘なし**になった(=L2:自律)」
- L2主張のコツ:「指摘なし」を偶然でなく再現性のあるプロセスとして語る。「レビューで過去に受けた指摘を自分用チェックリスト化し、セルフレビューしてから出すようにした」など、**指摘ゼロに至った仕組み**を説明できると強い。

**項目7(不具合修正の正しさ判断 L1→L2)← 業務①**
- 語り方:「修正が正しいと判断する自分なりの基準を持って運用した」として、①再現手順で直ったこと ②根本原因に対する修正であること(対症療法でない) ③影響範囲(同じパターンの他箇所・デグレ)を確認したこと、の3点セットで具体例を1つ語る。
- L2の根拠=「この3点を自分で設定し、レビュー依頼前に自己完結で確認していた」こと。

**項目8(不具合の切り分け -→L1)← 業務①②**
- 業務①:不具合対応で「担当機能の問題か、共通部品/他機能の問題か」を切り分けた具体例を1つ用意。
- 業務②:システムテストで検出した事象を「試験環境要因/構成ミス/プロダクトの不具合」に切り分けて有識者に繋いだ経験。「自分で解決できない課題は有識者に繋いだ」は、裏を返せば**繋ぐべきかどうかを切り分けた**ということ。ここを明示的に言語化する。

**項目9(高負荷結合試験 -→L1)← 業務②**
- 非機能試験の中身を「何の負荷を、どうかけて、何を観測し、合否をどう判定したか」の4点で1分で語れるようにする。L1(支援のもと)なので、手順書や有識者の支援を受けつつ完遂したことを正直に語ってよい。

**項目1・3への補強 ← 業務①のシーケンス図**
- 項目1:シーケンス図はUML振る舞い図。「UML表記を実務で使い、レビューを通した」実績として補強に使う(構造図=クラス図・構成図は成果物で埋める)。
- 項目3:再レビュー2回→OKの過程で「どんな観点の指摘を受け、何を直したか」を語る。指摘観点(例:異常系の欠落、ライフラインの粒度、実装との乖離)を**自分のレビューチェックリストに昇華した**話に繋げると、項目3「正当性確認ができる」の証拠になる。

### b) 業務外で成果物を作って証明すべき項目

項目1・2・3・5・6(+4・7・8・14の補強)。→ 第2章の成果物設計で一括カバー。

### c) 非技術系:行動・エピソードで語る項目

項目10・11・12・13・14・15。→ 第3章で語り方と「今からできる小さな行動」を提示。

## 2. 成果物設計:「車両DTC(故障コード)ログ解析ツール」

### 2-1. なぜこの題材か(面談でそのまま語れる理由)

- **業務ドメイン直結**:車載ソフトの不具合対応・システムテストという半期の業務と地続き。「単なるWebページ制作」にならない。
- **項目8(切り分け)がツールのテーマそのもの**:「どのECU/機能ドメインの不具合かを切り分ける」ツールなので、作る過程=切り分けの観点の言語化になる。
- **小さく作れて、設計・テスト・トレーサビリティの全工程を1周できる**:C#コンソールアプリ+クラスライブラリ+xUnitで、2週間のスキマ時間に収まる規模。
- **言語がC#**:項目4「開発で指定された言語」=業務で使うC#に合わせられる。

### 2-2. ツール仕様(最小構成)

- 入力:疑似DTCログ(CSV)。列:タイムスタンプ, ECU名, DTCコード, ステータス
- 処理:①パース+入力検証 → ②DTCコード→機能ドメイン対応ルールで分類 → ③「どの担当機能の不具合か」を切り分けたサマリ生成
- 出力:テキスト/Markdownレポート
- 規模目安:クラス6〜8個、実装400〜600行、テスト300行程度

### 2-3. アーキテクチャ(項目1・2の証明ポイント)

レイヤ構成(モジュール構成図に描く):
```
CLI層(Program) → Application層(DiagnosticAnalyzer)
              → Domain層(DtcRecord, FaultDomain, 分類ルール)
              → Infrastructure層(CsvLogParser, ReportWriter)
```
- 抽象:`ILogParser` / `IDtcRuleRepository` / `IReportWriter` をApplication層が参照(**DIP**)。Infrastructure層が実装。
- クラス図には:継承/実現/集約/依存の使い分け、インターフェイス、主要メソッドシグネチャを明記。
- 作図ツール:**draw.io(diagrams.net デスクトップ版)またはPlantUML/Mermaid**。いずれもPNG/SVG/PDFへ静的エクスポート可能=持ち込み条件を満たす。

**設計技法の選定理由書(項目2をL2にする核)**:A4半ページで以下を書く。
- 「入力→変換→出力の一本道なら構造化設計(DFD+モジュール分割)でも成立する。しかし本ツールは**分類ルールの追加・ログ形式の追加が予見される**ため、変更点を差し替え可能にするOO設計(インターフェイス+DIP)を選定した」
- 「テスト容易性:I/Oを抽象化することでApplication/Domain層を純粋な単体テスト対象にできる」
- **構造化設計を捨てた理由まで書けて初めてL2(自律的な技法選定)**。

### 2-4. 詳細設計の正当性確認(項目3)

- 業務①の再レビュー指摘から**自分のレビュー観点チェックリスト**を作る(例:異常系網羅/命名と責務の一致/図と要求のトレース/依存方向の妥当性/境界値の扱い)。
- そのチェックリストで成果物の設計書をセルフレビューし、**指摘→修正の記録(Before/After)を1〜2件残す**。「確認できる」ことの証拠は、確認して直した痕跡。

### 2-5. コーディングルール(項目4補強)

- `.editorconfig` + Microsoft C# コーディング規約 + Roslynアナライザを有効化し、**警告ゼロのビルドログ**をスクリーンショットで保存。
- 「業務の指摘なしマージ」+「業務外でもルールを自分で設定・遵守」の二段構えでL2を主張。

### 2-6. ホワイトボックステスト(項目5をL2に)

- 分類ロジックに意図的に分岐を持たせ(未知DTC、重複、時刻逆転、空ファイル等)、**判定条件ごとにテストケース表**(条件/入力/期待値/狙うカバレッジ)を作る。
- xUnit + coverlet で計測、**ReportGenerator で静的HTMLのカバレッジレポート**を出力(これがそのまま持ち込み資料になる)。
- 目標:対象ロジックの**C1(分岐網羅)100%**。ただし「C1 100%でも漏れるバグ(例:複合条件の組合せ、境界値のオフバイワン)」を1つ言えるように準備 → ここが「カバレッジを考慮できる=数字の限界も分かっている」というL2の差別化点。

### 2-7. 設計との整合性評価(項目6)

- **トレーサビリティマトリクス**を1枚作る:クラス図の要素 ↔ ソースファイル ↔ テストケース。
- 「設計と実装が乖離しやすい箇所(privateメソッドの増殖、設計にない依存)をどうチェックするか」を口頭で言えるようにする。

### 2-8. バグ注入演習(項目7・8の補強)

- 実装完了後、**意図的にバグを1つ仕込む**(例:境界条件の`<`と`<=`ミス)。
- 「事象再現 → ログ確認 → 層の切り分け(パース層か分類層か) → 原因特定 → 修正 → 回帰テスト全パス」の手順を**切り分け手順書(1ページ)+短い録画**に残す。
- 面談では「業務の不具合対応で使っている思考手順を、自分の題材で再現・言語化したもの」と説明。

### 2-9. AI利用ログ(項目14の証拠)

制作全体を通じて**AI利用ログ(表1枚)**を付ける:
| 日付 | AIに依頼した内容 | 出力の問題点(自分が発見) | 自分の修正・判断 |
- 「AIの初稿のクラス設計が○○の点で要件に合わず、△△に直した」「存在しないAPIを提示された(ハルシネーション)ので公式ドキュメントで裏取りした」等を最低3件記録。
- 機密情報を入れないルールを自分に課したことも書く(業務②のセキュリティルール展開経験と繋がる)。

### 2-10. 会社PCへの持ち込み方法(個人アカウントログインなし)

**最初にやること(Day 1):上長または情シスに「業務外学習の成果物を面談で見せたい。持ち込みはどの経路が許可されるか」を確認する。**
この確認行動自体が、項目13・14で語れるセキュリティ意識の証拠になる。そのうえで、許可される経路に応じて:

1. **静的ファイル一式(推奨)**:設計書・図はPDF/PNG、カバレッジレポートはReportGeneratorのHTML(フォルダごと)、コードはVS Codeの拡張(Print/Export系)やPandocで**シンタックスハイライト付きHTML/PDF化**。→ 会社で許可されたUSBメモリ or 許可されたファイル受け渡し経路で持込。
2. **デモは事前録画**:実行→バグ注入→切り分け→修正→テスト全パスの流れを**3〜5分のmp4**に録画(OBS等)。当日はライブ実行不要で事故らない。
3. **公開GitHubリポジトリの「閲覧」**:公開リポジトリの閲覧はログイン不要なので、ネットワーク的にgithub.comが閲覧可なら会社PCのブラウザで見せられる。ただし①公開して問題ない内容のみ(会社情報・業務コードを一切含めない)②社内ルールで問題ないか事前確認、の2条件必須。迷うなら方法1・2に倒す。
4. **前日リハ必須**:会社PCで全ファイルが開けるか、動画が再生できるかを面談前日に必ず確認(Day 13)。

### 2-11. 理解度チェックポイント(各ステップの合格基準)

| ステップ | 「これが説明できれば理解できている」 |
|---|---|
| 構成図・クラス図 | 「なぜこの層分割か」「矢印(依存)の向きが逆だと何が困るか」を図なしで口頭説明できる |
| 設計技法選定 | 「構造化設計で作るとどうなるか」を対比で語れる。「DIPを外したら何が壊れるか」に即答できる |
| 正当性確認 | チェックリストの各観点について「業務のどの指摘から来たか」を言える |
| 実装 | 任意のクラスを指されて、責務・依存先・対応する設計要素を10秒で答えられる |
| WBテスト | 「このテストを消すとカバレッジがどこで落ちるか」「C1 100%でも漏れるバグ例」を言える |
| 整合性評価 | マトリクスの任意の行で、設計↔コード↔テストを実ファイルで辿って見せられる |
| バグ注入演習 | 切り分け手順を、資料を見ずにホワイトボードで再現できる |
| AI利用 | AIの誤りを自分で直した実例を3件、具体的に語れる |

## 3. 非技術項目の語り方と「今からできる小さな行動」

### 項目12(チーム作業意識の向上 L1→L2)← 勤務態度問題への対応

L2=「自律的に」なので、**上司の指示を待たず、自分で判断して行動を完結させた**構図で語る。

> 「拠点の社員が自分一人という状況で、協力会社メンバーの居眠りが常習化しており、チームの士気と品質に影響すると判断しました。まず自分で直接指導しましたが改善せず、**個人への注意では解決しない構造的な問題**と判断し、チーム上長と社内担当部署を通じて原籍会社と正式にコミュニケーションを取るルートを取りました。結果、勤務態度は改善し、拠点全体の作業意識が引き締まりました。」

突っ込み対策:「なぜ直接指導で粘らなかった?」→「指導は2回(※実際の回数)行い改善が見られなかった時点で、雇用関係のない自分が続けても効果が薄く、むしろ関係悪化リスクがあると判断した。**エスカレーションは丸投げではなく、経緯・指導記録を整理して渡した**」——この「判断の理由」を語れることがL2の証明。

### 項目13(チーム方針の落とし込み -→L1)← 方針・セキュリティルール展開

- 「唯一の社員として、チーム方針とセキュリティルールを協力会社メンバーに落とし込んだ」を、**①何を(方針・ルールの中身の例1つ)②どうやって(口頭だけでなく資料化?朝会?)③浸透をどう確認したか(質問対応、行動の変化)**の3点セットで具体化。
- 「落とし込めた」証拠=メンバーの行動が変わった実例を1つ(例:ルールに沿った持ち出し申請が自発的に出るようになった等)。

### 項目10(客先からの信頼 -→L1)

直接の客先接点が薄い場合、正直さを保ちつつ2段構えで:
1. **半期の実績の再解釈**:試験結果の報告品質(期限遵守率、1日超過時の早期報告とリカバリ)は、発注元・上位部署=「お客様」への信頼行動として語れる。「遅延を隠さず即日報告し、リカバリ計画を提示した」は信頼の話。
2. **今からの小さな行動**:試験報告書のテンプレ改善案を1つ作って上長に提案する/客先向け報告に同席・レビュー参加を申し出る。「L1=支援のもとでできる」なら、**信頼獲得に向けた行動を開始した実績**で足りる。
- 注意:実体験が本当にゼロなら無理にL1を主張せず「来期この行動で到達する」と計画で語る方が、他項目の主張の信頼性を守れる。

### 項目11(業界情報で案件獲得に動く -→L1)

- 今から:**業界動向調査メモ(A4 1〜2枚)**を作る。題材例:OEM各社のSDV・AI戦略(自分が追っている情報でよい)、それが自社の担当領域(地図・診断・品質)にどう波及し、**どんな案件ニーズが生まれるか**まで書く。
- 上長かチームに共有し、「次の提案・案件でこの観点が使えるのでは」と一言添える。「案件獲得**に向けて動く**」のL1はこの一歩で立証できる。

### 項目14(AI活用の利点と注意点 -→L1)

- 成果物のAI利用ログ(2-9)がそのまま証拠。語りの型:「利点=初稿生成と壁打ちで速度が上がる。注意点=①ハルシネーション(実例)②設計判断を委ねると根拠が説明できなくなる③機密情報の入力禁止。だから**AIは下書き、判断とレビューは自分**という運用をした」。

### 項目15(チームのAI利用推進 -→L1)

- 今から:AI利用ログとチェックリスト(機密NG/裏取り必須/判断は人間)を**チームwiki・共有フォルダ向けのナレッジ資料1枚**に整え、上長に「チームに共有したい」と提案。可能なら5分のミニ勉強会を申し出る。
- 面談では資料を画面共有し「共有済み(or 共有を提案中)」と事実ベースで語る。

## 4. 想定Q&Aと回答例

**Q1. 生成AIを使ったんでしょ?本当に自分の力?**
> 「はい、AIは使いました。評価項目14・15にある通り、AIを正しく使えること自体が求められるスキルだと考えています。ただし役割分担を明確にしていて、設計判断・レビュー・テスト設計は自分、AIは初稿と壁打ちです。実際、AIの出力をこう直しました(AI利用ログの実例を提示)。中身は今この場でどこを聞かれても説明します。」
→ その場で「このクラスの責務は?」「この行を消すとどのテストが落ちる?」に即答して見せるのが最強の回答。

**Q2. 業務外の題材で、実務でどう活かせるの?**
> 「題材自体を業務(不具合の切り分け)から取っています。ここで言語化した切り分け手順書とレビューチェックリストは、来期のC#/WPF不具合対応と試験業務にそのまま持ち込みます。特に設計↔実装のトレーサビリティの観点は、既存コードの改修影響調査に使えます。」

**Q3. シーケンス図はなぜ再レビューが2回も必要だったのか?**
> 「初回は○○(例:異常系の欠落)、2回目は△△(例:実装との粒度不一致)の指摘を受けました。裏を返すと、レビューで見られる観点を2段階で学べたということで、その観点を自分のチェックリストに落とし、今回の成果物ではセルフレビューで事前に潰しています(チェックリストを提示)。」
→ 失敗の言い訳ではなく「レビュー観点の獲得プロセス」として語る。項目3の証拠に転換する。

**Q4. 試験の期限を1日超過した件は?**
> 「原因は○○(事実)。判明した時点で当日中に上長へ報告し、リカバリ計画を提示して翌日完了させました。以降は△△(例:中間チェックポイントの前倒し)で再発を防ぎ、残りの試験は期限内に完了しています。」

**Q5. 項目4や7で『L2=自律的』と言えるのはなぜ?**
> 「L2の定義を『方針立案から完了確認まで自分で判断して完結できる』と捉えています。最後の1か月は実装方針の立案→実装→セルフレビュー→指摘なしマージまで自分で回しました。不具合修正も、再現・根本原因・影響範囲の3点を自分の基準として運用していました。」

**Q6. 一人拠点で、判断に迷ったときどうしていた?**
> 「自分で解決できる課題か、有識者に繋ぐべき課題かをまず切り分け、繋ぐ場合も丸投げせず状況を整理して渡していました(具体例1つ)。」→ 項目8と12を同時に補強できる。

**Q7. カバレッジ100%ならバグはないと言える?**(項目5の引っかけ)
