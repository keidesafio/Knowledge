# バグ注入演習:切り分けの思考プロセスを見せる3パターン

評価項目7「不具合修正の正しさ判断(L2)」・項目8「不具合の切り分け(L1)」の立証材料。
3つは**切り分けの軸が異なる**ように選んである:

| # | バグの所在 | 切り分けの軸(面談で見せる問い) |
|---|---|---|
| ① | Infrastructure層(パース) | 「**どのデータで**壊れるか」→ データパターンから層を特定する |
| ② | ルールデータ(コード外) | 「**コードかデータか**」→ 同じ症状でも原因の置き場所が違う |
| ③ | 層間の契約 | 「**単体か結合か**」→ 単体で正しいのに結合で壊れる |

運用:各バグは `bug/1-parse-format` のように**専用ブランチ**に仕込み、切り分け→修正の過程をコミット履歴で残す。デモ録画もブランチ単位で撮る。

---

## パターン①:日時フォーマットの MM/mm 取り違え(パース層・データ依存)

### 仕込み
```csharp
// CsvLogParser 内。正:"yyyy-MM-dd HH:mm:ss" 誤:分(mm)の位置に月(MM)
DateTime.TryParseExact(fields[0], "yyyy-MM-dd HH:MM:ss",   // ★MMは「月」
    CultureInfo.InvariantCulture, DateTimeStyles.None, out var ts)
```

### 症状
一部の行だけ ParseErrors に落ちる。全滅ではないのが肝:**分が01〜12の行は「月」として解釈できてしまい成立**、13分以降の行だけ失敗する。

### 発見
レポートの ParseErrors 件数が想定より多い/実行のたびに「エラーになる行の顔ぶれ」に規則性がある。

### 切り分け手順(思考の順に)
1. **再現の固定**:エラーになった行を3行抜き出し、その3行だけの最小CSVで再現することを確認(「たまに起きる」を「必ず起きる」に変換するのが第一歩)。
2. **層の特定**:症状は「行が読めない」なので、分類・集計より手前のパース層と仮説。`CsvLogParser` 単体テストに問題の行を直接入力 → 単体で再現 → **Infrastructure層内に閉じた不具合**と確定(Analyzer・Domainは容疑から外す)。
3. **パターンの抽出**:失敗行と成功行を並べて差分を探す。「失敗行はすべて分が13以上」という規則を発見 → 「分のフィールドが月として解釈されている?」という仮説に到達。
4. **原因確定**:フォーマット文字列を目視 → `HH:MM:ss` を発見。MM=月/mm=分の仕様を公式ドキュメントで確認(推測で直さない)。
5. **修正**:`HH:mm:ss` に修正。
6. **修正の正しさ判断(項目7の3点セット)**:①再現ケース(分=13以降の行)が通る ②根本原因への修正である(エラー行をスキップする等の対症療法ではない)③回帰:全テスト緑+**「分が00/12/13/59の行」を境界テストとして追加**し再発を防止。

### 面談での語りポイント
「全滅ではなく一部だけ壊れる不具合は、**壊れるデータと壊れないデータの差分**が最短の手がかりになる。業務のログ解析でも同じで、事象の発生条件のパターン化から入ります。」

## パターン②:到達不能ルール(コードは無傷、データが原因)

### 仕込み
```csharp
// InMemoryDtcRuleRepository のルール定義(コードのロジックは一切触らない)
new("P0",  FaultDomain.Powertrain),
new("P05", FaultDomain.Chassis),   // ★先勝ちのため永遠に到達しない
```

### 症状
"P0562"(本来Chassis)がPowertrainに分類される。**単体テストは全緑・C1カバレッジ100%のまま**(前回文書のパターン③の実演)。

### 発見
レポートのドメイン別集計が既知の期待値と合わない(Chassisが0件)。あるいは本番ルールセットを流す「ルール網羅テスト」を追加した瞬間に赤くなる。

### 切り分け手順
1. **再現の固定**:"P0562" 1件だけの最小入力で誤分類を再現。
2. **コードかデータかの二分**:同じ `DtcClassifier` に**最小ルールセット** `[("P05", Chassis)]` だけを与えて "P0562" を分類 → 正しくChassisになる → **分類コードは正常**。次に本番ルールセットを与える → 誤る。⇒ 原因は**ルールデータ側**と確定。この「コードを固定してデータを差し替える/データを固定してコードを差し替える」対照実験が切り分けの核。
3. **原因確定**:ルールの並びを見て、先勝ち仕様と "P0" が "P05" を包含する関係に気づく。
4. **修正の選択肢を比較(ここが項目7のL2ポイント)**:
   - 案a:並び順を直す(P05を先に)——最小修正だが、**同種バグの再発は防げない**。
   - 案b:読み込み時に「より長いプレフィックスを先に」ソートする——仕組みで再発防止。
   - 案c:到達不能ルールを検出して警告するバリデーション追加——検出の仕組み化。
   採用:b+c(aだけでは根本対応でないと判断した理由を語る)。
5. **正しさ判断**:再現ケース修正+ルール網羅テスト(全ルールに代表DTC1件)を追加して全緑。

### 面談での語りポイント
「テストが全部通っているのに出力がおかしい、はコード以外(データ・設定・環境)を疑うサイン。車載でいえば**ソフト不具合かパラメータ/コンフィグの問題かの切り分け**と同型です。担当機能の不具合かどうかを判断する(項目8)とは、この対照実験を組めることだと考えています。」

## パターン③:層間の契約解釈違い(単体は両方正しい、結合で壊れる)

### 仕込み
Unknownステータス行の扱いについて、両層が**別々の解釈**をしている状態を作る:
```csharp
// CsvLogParser:未知ステータスは Unknown としてレコードに含め、かつ Errors にも記録
records.Add(rec with { Status = DtcStatus.Unknown });
errors.Add($"line {n}: unknown status '{raw}' (kept as Unknown)");

// DiagnosticAnalyzer:TotalRecords = records.Count + errors.Count  ★エラー=除外行と解釈
```

### 症状
レポートの `TotalRecords` が入力行数より多い(**二重計上**)。ドメイン別集計の合計とも一致しない。

### 発見
「集計の内訳合計 = TotalRecords = 入力行数」を検証する**整合性アサーションを持つ結合テスト**が赤くなる。単体テストはパーサもAnalyzerも緑(それぞれ自分の解釈では正しいから)。

### 切り分け手順
1. **再現の固定**:未知ステータス1行を含む3行CSVで、期待3件・実際4件を確認。
2. **単体か結合かの二分**:パーサ単体テスト緑、Analyzer単体テスト(フェイクパーサ注入)緑、結合だけ赤 → **どちらのクラスの中でもなく、間(契約)の問題**と切り分ける。単体が両方緑の時点で「犯人捜し」から「解釈のズレ探し」に問いを切り替えるのがポイント。
3. **契約の突き合わせ**:`ParseResult` の設計書定義を確認 → 「Errorsは*除外した行*の記録」なのか「*警告*も含む」のか、実は設計書に明記がなかったことが判明(**不具合の根本原因は実装ミスでなく仕様の曖昧さ**、という着地)。
4. **修正**:契約を明文化して片側を直す。採用案:Errorsは「レコード化できなかった行」のみとし、Unknown化は `AnalysisResult` に警告件数として別枠で出す。設計書(ParseResultのXMLドキュメントコメント)にも追記。
5. **正しさ判断**:整合性アサーション(内訳合計=Total=入力行数)を結合テストに恒久化。**同じ曖昧さが他のインターフェイスにないか横展開で点検**(IReportWriterのnull要素の扱い等)——「他に同じパターンはないか」まで見るのがL2。

### 面談での語りポイント
「システムテスト(業務②)で単体試験済みの機能同士を結合したときの不具合は、まずインターフェイスの解釈差を疑っていました。この演習はそれを自分のコードで最小再現したものです。修正がコードでなく**設計書の明文化**に及んだ点が、一番実務に近い学びでした。」

## 共通:切り分けの型(3パターンを貫く手順として面談冒頭に示す)

1. **再現を固定する**(最小入力に削る。「必ず起きる」にしてから考える)
2. **二分する問いを立てる**(どの層か/コードかデータか/単体か結合か)
3. **対照実験で片側を消す**(1要素だけ差し替えて挙動を比較)
4. **原因を仕様・ドキュメントで裏取りしてから直す**(推測で直さない)
5. **修正の正しさを3点で判断**(再現ケース解消/根本原因対応/回帰+再発防止テスト追加)
