## A1. 設計原則 ― 依存の方向を制御する技術

### ① なぜミドルに必要か

現場でこういう場面が来る。「DTCログの入力元をCSVからCANバス直結に変えてほしい。ただし解析ロジックと画面は一切触るな」。ジュニアはこの依頼で解析クラスの中のファイル読み込み部分を書き換えて回る。ミドルは最初から入力元を差し替え可能な構造にしておくので、新しいクラスを1つ足すだけで済む。この差は「知っているか」ではなく「依存の方向を設計時に制御したか」の差。レビューで「このクラス、リポジトリの具象に依存してますよ」と指摘できるのがミドルの入り口。

### ② 本質的な概念

本質は1文で言える。**「変更されやすいもの(具象)に、変更されにくいもの(ビジネスロジック)を依存させない」**。これがDIP(依存性逆転の原則:上位モジュールは下位モジュールの実装ではなく抽象に依存せよ、という原則)。

**初学者の誤解と正しい理解の対比:**

| 誤解 | 正しい理解 |
|------|-----------|
| 「DIコンテナを使う=DIを理解した」 | DIコンテナは道具。本質はコンストラクタで依存を外から渡すこと。コンテナなしでも `new DtcAnalyzer(new CsvRepository())` と手で組めばDIは成立する |
| 「とりあえず全クラスにインターフェースを切る」 | 実装が1つしかなく差し替え予定も無いものにインターフェースを切るのはノイズ。切るのは「変更の境界」(I/O、外部サービス、時刻取得など)だけ |
| 「SOLIDを暗記した=設計できる」 | SOLIDは結果論。先にあるのは「このコード、何が変わったら壊れるか?」という問い。答えた結果がSOLIDの形になる |
| 「疎結合=クラスを細かく分ける」 | 疎結合とは「クラスAを修正したときクラスBに一切触れなくて済む状態」。細かく分けても互いの内部を知っていれば密結合 |

### ③ 言語横断の視点

**変わらない本質:**「抽象(契約)に依存し、具象は外から注入する」。これはC#でもPythonでもRustでも同じ。

**言語固有の実装の違い:**

| 観点 | C# | Python |
|------|-----|--------|
| 抽象の表現 | `interface`(コンパイラが実装漏れを検出) | `Protocol`(構造的部分型:メソッドの形が合っていれば暗黙的に適合)または`ABC`(明示的継承を要求) |
| 適合チェックのタイミング | コンパイル時 | 実行時(mypy等の型チェッカを使えば静的にも検査可能) |
| 注入の慣習 | DIコンテナ(Microsoft.Extensions.DependencyInjection)が標準的 | コンテナを使わず素のコンストラクタ注入が主流。フレームワーク(FastAPIの`Depends`等)が持つ場合もある |

重要な気づき:**PythonのProtocolはC#のinterfaceより「ゆるい契約」**。C#では`class CsvRepo : IDtcRepository`と明示宣言が必要だが、Protocolは`load_all`メソッドを持ってさえいれば適合扱いになる(ダックタイピングに型を付けたもの)。この「明示 vs 構造」の違いを知ると、Goのinterfaceも同じ構造的型付けだと一発で理解できる。

### ④ コード例:DTCログ解析の入力元差し替え

**C#:**

```csharp
public record DtcRecord(string Code, DateTime Timestamp, int Severity);

// 変更の境界にだけインターフェースを切る
public interface IDtcRepository
{
    IReadOnlyList<DtcRecord> LoadAll();
}

public class CsvDtcRepository : IDtcRepository
{
    private readonly string _path;
    public CsvDtcRepository(string path) => _path = path;

    public IReadOnlyList<DtcRecord> LoadAll() =>
        File.ReadLines(_path)
            .Skip(1)
            .Select(line => line.Split(','))
            .Select(c => new DtcRecord(c[0], DateTime.Parse(c[1]), int.Parse(c[2])))
            .ToList();
}

// 解析ロジックは抽象にしか依存しない → 入力元が変わっても無傷
public class DtcAnalyzer
{
    private readonly IDtcRepository _repo;
    public DtcAnalyzer(IDtcRepository repo) => _repo = repo;  // コンストラクタ注入

    public IEnumerable<DtcRecord> FindCritical(int threshold) =>
        _repo.LoadAll().Where(r => r.Severity >= threshold);
}
```

**Python(Protocol版):**

```python
from dataclasses import dataclass
from datetime import datetime
from pathlib import Path
from typing import Protocol


@dataclass(frozen=True)
class DtcRecord:
    code: str
    timestamp: datetime
    severity: int


class DtcRepository(Protocol):          # C#のinterfaceに相当。ただし構造的型付け
    def load_all(self) -> list[DtcRecord]: ...


@dataclass
class CsvDtcRepository:                 # 継承宣言は不要。load_allを持つだけで適合
    path: Path

    def load_all(self) -> list[DtcRecord]:
        records = []
        for line in self.path.read_text().splitlines()[1:]:
            code, ts, sev = line.split(",")
            records.append(DtcRecord(code, datetime.fromisoformat(ts), int(sev)))
        return records


class DtcAnalyzer:
    def __init__(self, repo: DtcRepository) -> None:   # コンストラクタ注入は完全に同型
        self._repo = repo

    def find_critical(self, threshold: int) -> list[DtcRecord]:
        return [r for r in self._repo.load_all() if r.severity >= threshold]
```

CANバス直結に変える日が来たら、`CanBusDtcRepository`を追加するだけ。`DtcAnalyzer`は1文字も変わらない。これが「依存の方向を制御した」状態。

### ⑤ 確認問題

1. 上のC#コードで`DtcAnalyzer`が`CsvDtcRepository`を直接`new`した場合、単体テストで何が困るか。具体的なテストコードの困りごととして説明せよ。
2. `DateTime.Now`(Pythonなら`datetime.now()`)を直接呼ぶクラスはテストしづらい。「変更の境界」の考え方でこれをどう設計し直すか、C#とPython両方でコードを書け。
3. PythonのProtocolとABCの使い分け基準を1つ挙げ、「自作ライブラリを外部に公開する場合」はどちらが適切か理由付きで答えよ。

## A2. テスト設計 ― 「壊れたら即分かる」状態を作る技術

### ① なぜミドルに必要か

現場のリアルな場面:リリース2日前に「DTC重要度の判定条件を1つ追加して」と言われる。テストが無いコードベースでは、修正後に「他が壊れていないこと」を手動で全画面確認する羽目になり、2日では終わらない。テストがあれば修正→全テスト実行(3分)→緑なら出荷、で終わる。**テストは品質保証の道具である前に、「変更のスピードを維持する道具」**。ミドルの評価軸は「速く安全に変更できるか」であり、その土台がテスト。さらに車載領域ではISO 26262がテスト成果物を要求するので、テスト設計力はドメイン適合性が特に高い。

### ② 本質的な概念

良いテストの定義は「**実装の詳細ではなく、観測可能な振る舞いを検証するテスト**」。ここを外すと、リファクタのたびにテストが壊れる「壊れやすいテスト」を量産する。

**初学者の誤解と正しい理解の対比:**

| 誤解 | 正しい理解 |
|------|-----------|
| 「カバレッジ100%が良いテスト」 | カバレッジは「テストされていない場所」の検出には使えるが、高くても振る舞いを検証していなければ無意味。アサーションの無いテストでもカバレッジは上がる |
| 「privateメソッドもテストすべき」 | privateはテストしない。publicの振る舞い経由で間接的に検証する。privateを直接テストしたくなったら、それは別クラスに抽出すべきサイン |
| 「モックをたくさん使うのが上級者」 | モックの多用は実装詳細への結合を生む。優先順位は 本物 > フェイク(簡易実装)> スタブ > モック。モックは「外部との相互作用自体が仕様」の時だけ(例:通知を1回だけ送ったことの検証) |
| 「テストは実装が終わってから書く」 | 少なくとも「テストしやすい形か?」は実装前に考える。テストしにくいコードは設計が悪いことのアラート(A1と直結) |

テストケースの導出には**境界値分析**(条件の境目の値を狙う:閾値が3なら2, 3, 4を試す)と**同値分割**(同じ振る舞いになる入力群から代表1つを選ぶ)を使う。これはFMEAで故障モードを列挙する思考と同型で、「どこで壊れうるか」を系統的に洗い出す技術。

### ③ 言語横断の視点

**変わらない本質:**AAA構造(Arrange準備→Act実行→Assert検証)、境界値分析、テストダブルの使い分け、「振る舞いをテストする」原則。テスト設計の思考は100%言語非依存。

**言語固有の違い:**

| 観点 | C# | Python |
|------|-----|--------|
| 標準的FW | xUnit / NUnit | pytest |
| テストの単位 | クラス+属性(`[Fact]`, `[Theory]`) | 素の関数+命名規約(`test_`で始まる関数) |
| パラメタライズ | `[Theory] + [InlineData]` | `@pytest.mark.parametrize` |
| テストダブル | Moq等のライブラリでinterfaceから動的生成 | Protocolのおかげで**手書きフェイクが3行で書ける**。`unittest.mock`もあるが手書きフェイク優先 |
| 前処理の共有 | コンストラクタ/`IClassFixture` | fixture(`@pytest.fixture`で定義し引数名で注入) |

Pythonのfixtureは「引数名でDIされる」独特の仕組みで、C#経験者が最初に戸惑うポイント。だが「テストにも依存を注入する」という見方をすればA1と同じ原則だと分かる。

### ④ コード例:A1のDtcAnalyzerをテストする

**C#(xUnit、手書きフェイク):**

```csharp
// フェイク:本物の代わりの簡易実装。interfaceがあるから3行で書ける
public class FakeDtcRepository : IDtcRepository
{
    private readonly IReadOnlyList<DtcRecord> _records;
    public FakeDtcRepository(params DtcRecord[] records) => _records = records;
    public IReadOnlyList<DtcRecord> LoadAll() => _records;
}

public class DtcAnalyzerTests
{
    [Theory]
    [InlineData(2, 2)]  // 境界値: 閾値ちょうど → 含む
    [InlineData(3, 1)]  // 境界値: 閾値+1のみ該当
    public void FindCritical_閾値以上のみ返す(int threshold, int expectedCount)
    {
        // Arrange
        var repo = new FakeDtcRepository(
            new DtcRecord("P0301", DateTime.Now, 1),
            new DtcRecord("P0420", DateTime.Now, 2),
            new DtcRecord("U0100", DateTime.Now, 3));
        var sut = new DtcAnalyzer(repo);   // sut = System Under Test(テスト対象)

        // Act
        var result = sut.FindCritical(threshold);

        // Assert
        Assert.Equal(expectedCount, result.Count());
    }
}
```

**Python(pytest):**

```python
import pytest
from datetime import datetime
from dtc import DtcAnalyzer, DtcRecord


class FakeDtcRepository:                       # 継承宣言すら不要(Protocolの恩恵)
    def __init__(self, *records: DtcRecord) -> None:
        self._records = list(records)

    def load_all(self) -> list[DtcRecord]:
        return self._records


@pytest.fixture
def repo() -> FakeDtcRepository:
    now = datetime.now()
    return FakeDtcRepository(
        DtcRecord("P0301", now, 1),
        DtcRecord("P0420", now, 2),
        DtcRecord("U0100", now, 3),
    )


@pytest.mark.parametrize(("threshold", "expected"), [(2, 2), (3, 1)])
def test_find_critical_returns_only_at_or_above_threshold(
    repo: FakeDtcRepository, threshold: int, expected: int
) -> None:
    sut = DtcAnalyzer(repo)                    # Arrange
    result = sut.find_critical(threshold)      # Act
    assert len(result) == expected             # Assert
```

2つを並べると、フレームワークの見た目は違っても「フェイクを注入し、境界値で振る舞いを検証する」構造が完全に同じだと分かる。

2. `DtcAnalyzer`のテストで`CsvDtcRepository`+実ファイルを使う方式と、フェイクを使う方式のトレードオフを「実行速度」「壊れる原因の特定しやすさ」「検証できる範囲」の3軸で比較せよ。

## A3. 問題分解とデバッグ戦略 ― 未知の不具合を系統的に潰す技術

### ① なぜミドルに必要か

「たまに画面が固まるらしい。再現手順は不明」――この種のチケットが来たとき、ジュニアは手当たり次第にコードを眺めて時間を溶かす。ミドルは**仮説→実験→絞り込み**のループを回して、2時間で原因空間を1/10に狭める。デバッグ速度は経験年数ではなく手順の有無で決まる。そして車載ドメインでは、これはまさにフォールトアイソレーション(故障箇所の切り分け)であり、FMEAの「故障モードから原因を遡る」思考の実践版。ここを言語化できると、DTC解析ツールのような成果物の説明にも直結する。

### ② 本質的な概念

デバッグの本質は**二分探索**。「原因はこの範囲のどこかにある」という空間を、1回の実験で半分に割る質問を繰り返す。

**初学者の誤解と正しい理解の対比:**

| 誤解 | 正しい理解 |
|------|-----------|
| 「コードを読めば原因が見つかる」 | 読むのは仮説ができてから。先に必要なのは再現条件の特定。「読む」は実験の一手段にすぎない |
| 「printデバッグは素人のやること」 | print(ログ)は立派な計測手段。悪いのは仮説なしに闇雲に仕込むこと。「この仮説が正しければこの値はXのはず」と予言してから仕込む |
| 「直った=解決」 | なぜその修正で直ったのか説明できないなら解決ではない。偶然直った修正は別の場所で再発する |
| 「バグは1つずつ潰す」 | 修正の前に最小再現コードを作る。再現が10秒で回れば実験速度が10倍になり、結果的に速い |

**手順の型(これを暗記するのではなく、毎回このシートを埋める):**

1. **観測**:期待した動作と実際の動作を1文ずつで書く(「固まる」ではなく「ボタン押下後3秒間UIスレッドが応答しない」)
2. **再現**:最小の再現手順を作る。再現率も記録(10回中3回、など)
3. **仮説**:原因候補を列挙し、「もしこれが原因なら、○○すると△△になるはず」と反証可能な形で書く
4. **実験**:1回の実験で候補空間が最も半分に近く割れるものから実施
5. **記録**:棄却した仮説も残す(翌日の自分と同僚のため)

コミット履歴に対する二分探索が`git bisect`(壊れたコミットを対数回のテストで特定するGitコマンド)。1000コミットでも10回のテストで特定できる。

### ③ 言語横断の視点

**変わらない本質:**仮説駆動、二分探索、最小再現、観測手段の確保。この思考プロセスはC/C++の組込みデバッグでもPythonのエージェント開発でも同一。

**言語固有の違い:**

| 観点 | C# | Python |
|------|-----|--------|
| 対話的デバッガ | Visual Studioデバッガ(条件付きブレークポイント、データブレークポイントが強力) | `breakpoint()`組込関数でpdb起動 / VS Codeデバッガ |
| 典型的な罠 | async void内の例外が握りつぶされる、UIスレッドのデッドロック(`.Result`ブロッキング) | 例外の握りつぶし(`except Exception: pass`)、ミュータブルなデフォルト引数、asyncioでのブロッキング呼び出し |
| 事後解析 | 例外のスタックトレース+dmpファイル | スタックトレース+`traceback`モジュール、`faulthandler` |
| 履歴の二分探索 | `git bisect`(共通) | `git bisect`(共通) |

言語が変わって差が出るのは「観測の道具」だけで、「何を観測すべきか決める思考」は共通。道具の使い方は都度調べればよいが、思考は事前に訓練が要る。

### ④ 設計例:デバッグシートの実例

「DTC解析ツールで、特定のログファイルだけ解析結果が0件になる」というバグを想定した実際の記入例:

```text
【観測】
期待: sample_20260315.log(1,204行)を解析すると重要度2以上が約40件出る
```

## B1. 並行・非同期処理の本質 ― 「待ち時間」と「計算」を区別する技術

### ① なぜミドルに必要か

エージェント開発で必ずこの場面が来る。「Anthropic APIを5回呼ぶ処理が25秒かかる。速くして」。ジュニアは「スレッドを増やす?」と考えるが、ミドルはまず「この25秒は待ち時間か計算か」を問う。API呼び出しはほぼ全部がネットワーク待ち(I/Oバウンド)なので、並行に待てば5秒に縮む。逆に画像処理のようなCPU計算(CPUバウンド)はいくら非同期にしても速くならない。この区別ができないと、効果のない最適化に日単位の工数を溶かす。C#のasync/awaitを半年使った今こそ、「なぜ動くのか」を言語非依存で理解し直す好機。

### ② 本質的な概念

本質:**非同期とは「待っている間、スレッドを他の仕事に貸し出す仕組み」であって、処理を速くする魔法ではない**。

**初学者の誤解と正しい理解の対比:**

| 誤解 | 正しい理解 |
|------|-----------|
| 「asyncを付けると速くなる」 | 1つのI/O処理単体は1ミリ秒も速くならない。速くなるのは「複数の待ちを重ねられる」場合のみ |
| 「async/await=マルチスレッド」 | 別物。asyncは「待ちの間スレッドを解放する」協調的な仕組み。マルチスレッドは「複数の実行の流れを同時に走らせる」仕組み。C#では両者が混ざって見えるので混同しやすい |
| 「awaitを付ければ並行になる」 | `await A(); await B();` は直列(Aが終わってからB)。並行にするには両方を先に開始してからまとめて待つ(`Task.WhenAll` / `asyncio.gather`) |
| 「非同期メソッドの中なら何をしてもいい」 | 非同期の文脈でブロッキング呼び出し(`Thread.Sleep`、Pythonの`time.sleep`や同期版`requests`)をすると、スレッド/イベントループ全体が止まる。最悪のアンチパターン |

判断の型:処理を見たらまず **I/Oバウンド(ネットワーク・ディスク・DB待ち)か、CPUバウンド(計算)か** を分類する。I/Oバウンド→非同期I/Oで並行化。CPUバウンド→マルチプロセス/別スレッド(C#なら`Task.Run`)。この分類が全ての出発点。

### ③ 言語横断の視点

**変わらない本質:**I/O待ちの間に他の仕事をするという発想、async/awaitという構文、「直列awaitと並行gatherの違い」、キャンセルの伝播という概念。

**言語固有の違い(ここはかなり大きい):**

| 観点 | C# | Python |
|------|-----|--------|
| 実行基盤 | スレッドプール上でタスクが動く。awaitの前後で**別スレッドになりうる** | 単一スレッドのイベントループ。awaitの前後も同じスレッド |
| CPU処理の逃し先 | `Task.Run`でスレッドプールへ | GIL(グローバルインタプリタロック:CPythonで同時に1スレッドしかバイトコードを実行できない制約)があるため、スレッドではCPU処理は並列化されない。`multiprocessing`や`concurrent.futures.ProcessPoolExecutor`でプロセスを分ける |
| 並行待ち | `Task.WhenAll(t1, t2)` | `asyncio.gather(c1, c2)` |
| ブロッキングの罠 | UIスレッドで`.Result`/`.Wait()`するとデッドロック | イベントループ内で`time.sleep`や同期`requests`を呼ぶと全コルーチンが停止 |
| キャンセル | `CancellationToken`を明示的に引き回す | `asyncio.CancelledError`が自動伝播 |

最大の違いは実行モデル:**C#は「スレッドプール+タスク」、Pythonのasyncioは「単一スレッド+イベントループ」**。C#の感覚で「awaitすれば裏で別スレッドが働く」と思い込むと、Pythonで同期ライブラリを混ぜて全停止させる事故を起こす。

### ④ コード例:API呼び出し5件を直列→並行にする

**C#:**

```csharp
// 直列: 約25秒(5秒×5)
public async Task<List<string>> AnalyzeSequentialAsync(List<string> logs)
{
    var results = new List<string>();
    foreach (var log in logs)
        results.Add(await CallClaudeAsync(log));   // 1件終わるまで次に進まない
    return results;
}

// 並行: 約5秒(全部同時に待つ)
public async Task<string[]> AnalyzeConcurrentAsync(List<string> logs)
{
    var tasks = logs.Select(log => CallClaudeAsync(log));  // 全部開始
    return await Task.WhenAll(tasks);                       // まとめて待つ
}
```

**Python:**

```python
import asyncio


# 直列: 約25秒
async def analyze_sequential(logs: list[str]) -> list[str]:
    results = []
    for log in logs:
        results.append(await call_claude(log))   # 1件ずつ待つ
    return results


# 並行: 約5秒
async def analyze_concurrent(logs: list[str]) -> list[str]:
    tasks = [call_claude(log) for log in logs]   # コルーチンを列挙
    return await asyncio.gather(*tasks)          # まとめて待つ

# なお実際のAPI呼び出しではレート制限があるので、同時実行数を絞る:
async def analyze_bounded(logs: list[str], limit: int = 3) -> list[str]:
    sem = asyncio.Semaphore(limit)               # 同時にlimit件まで
    async def _one(log: str) -> str:
        async with sem:
            return await call_claude(log)
    return await asyncio.gather(*[_one(log) for log in logs])
```

`Task.WhenAll`と`asyncio.gather`が完全に対応することを見れば、「概念は同じ、名前が違うだけ」の典型例だと分かる。SemaphoreによるスロットリングはC#では`SemaphoreSlim`で同型に書ける。

### ⑤ 確認問題

1. 「1万行のログをパースして統計を取る処理」と「10個のURLからログをダウンロードする処理」のそれぞれについて、I/O・CPUの分類と、C#/Python双方での適切な並行化手段を答えよ。
2. Pythonのasync関数の中で`time.sleep(5)`を呼ぶと何が起きるか。`await asyncio.sleep(5)`との違いを、イベントループの動きで説明せよ。
3. `asyncio.gather`で5件中1件が例外を投げた場合のデフォルト動作と、全件の結果(成功/失敗)を回収したい場合の書き方を調べて答えよ。

## B2. データモデリングと永続化 ― データの形が変わらない設計をする技術

### ① なぜミドルに必要か

コードは書き直せるが、**溜まったデータは書き直せない**。テーブル設計を間違えたまま半年運用すると、移行スクリプト+ダウンタイム+移行ミスのリスクという三重苦になる。現場では「DTCログを月100万件保存して、車両IDと期間で検索したい」のような要件が来る。ここで正規化・インデックス・トランザクションを理解していないと、動くが3ヶ月後に破綻する設計をしてしまう。またEF CoreやSQLAlchemyのようなORM(オブジェクトとテーブルを対応付けるライブラリ)は便利だが、裏で発行されるSQLを想像できないと、N+1問題(後述)で本番だけ激遅になる事故を起こす。

### ② 本質的な概念

本質は2つ。(1)**事実は1箇所にだけ書く(正規化)**。同じ事実が2箇所にあると、片方だけ更新されて矛盾する。(2)**一連の変更は「全部成功」か「全部なかったこと」のどちらかにする(トランザクション)**。

**初学者の誤解と正しい理解の対比:**

| 誤解 | 正しい理解 |
|------|-----------|
| 「正規化は理論の話、実務では気にしない」 | 逆。非正規化していいのは「正規化した上で、計測して遅かった箇所だけ」。順番が重要 |
| 「ORMを使えばSQLは知らなくていい」 | ORMはSQL生成器。生成されるSQLを読めない人がORMを使うと、ループ内クエリ(N+1)を量産する |
| 「インデックスは全カラムに張れば速い」 | インデックスは検索を速くする代わりに書き込みを遅くし、容量を食う。WHERE/JOIN/ORDER BYに使うカラムに絞る |
| 「トランザクション=DBが勝手にやってくれるもの」 | 境界(どこからどこまでを1つの作業単位にするか)を決めるのは設計者。「注文作成と在庫減算は同一トランザクション」という判断はコードに書く人の責任 |

**N+1問題**:「車両一覧を取得(1回)→各車両のDTCを車両ごとに取得(N回)」で合計N+1回のクエリが飛ぶ問題。開発環境(N=5)では気づかず、本番(N=10万)で爆発する。対策は「関連データを先読みする」1択で、あとはORMごとの書き方の問題。

### ③ 言語横断の視点

**変わらない本質:**正規化理論、トランザクションのACID特性、インデックスの仕組み(B木)、N+1問題とその対策方針。SQLそのものも(方言はあれど)ほぼ共通言語。

**言語固有の違い:**

| 観点 | C# (EF Core) | Python (SQLAlchemy 2.x) |
|------|--------------|--------------------------|
| クエリの書き方 | LINQ(`context.Vehicles.Where(...)`) | `select(Vehicle).where(...)` |
| 先読み(N+1対策) | `.Include(v => v.Dtcs)` | `options(selectinload(Vehicle.dtcs))` |
| 変更追跡 | ChangeTracker(自動) | Sessionのunit of work(自動) |
| マイグレーション | `dotnet ef migrations add` | Alembic |
| 遅延ロード | 既定で無効(明示的に有効化) | 既定で有効(relationshipのlazy設定)← ここがN+1の温床 |

EF CoreとSQLAlchemyは思想が近く(どちらもUnit of Workパターン)、EF Core経験者はSQLAlchemyの学習コストが低い。差が出るのは遅延ロードのデフォルト挙動で、SQLAlchemyは既定で遅延ロードのためN+1を踏みやすい。

### ④ コード例:N+1の発生と対策

**C# (EF Core):**

```csharp
// NG: N+1。車両100台ならクエリ101回
var vehicles = await context.Vehicles.ToListAsync();          // 1回
foreach (var v in vehicles)
{
    // 遅延ロードが有効だと、ここで毎回SELECTが飛ぶ
    Console.WriteLine($"{v.Vin}: {v.Dtcs.Count}件");           // N回
}

// OK: 先読み。クエリは実質2回に固定
var vehicles = await context.Vehicles
    .Include(v => v.Dtcs)                                      // JOIN/分割クエリで先読み
    .ToListAsync();
```

**Python (SQLAlchemy 2.x):**

```python
from sqlalchemy import select
from sqlalchemy.orm import selectinload

# NG: N+1。lazy="select"(既定)だと属性アクセス時に毎回SELECT
vehicles = session.scalars(select(Vehicle)).all()      # 1回
for v in vehicles:
    print(f"{v.vin}: {len(v.dtcs)}件")                  # ここでN回飛ぶ

# OK: selectinloadで先読み。クエリは2回に固定
stmt = select(Vehicle).options(selectinload(Vehicle.dtcs))
vehicles = session.scalars(stmt).all()
```

対策の検証方法も型にする:**発行SQLをログに出して数える**(EF Coreは`LogTo(Console.WriteLine)`、SQLAlchemyは`echo=True`)。「速くなった気がする」ではなくクエリ回数という数字で確認する。

### ⑤ 確認問題

1. 「DTCテーブルに車両のメーカー名カラムを持たせる」設計の問題点を、正規化の観点から具体的な更新シナリオ付きで説明せよ。
2. 自分のエージェント開発のDB(会話履歴保存など)で`echo=True`を有効にし、1画面の表示で発行されるSQL回数を数え、N+1があれば先読みに直したうえでbefore/afterの回数を記録せよ。
3. 「口座Aから口座Bへ送金」でトランザクションを張らなかった場合に起きうる不整合を、途中失敗のタイミング別に2パターン挙げよ。

## B3. コードリーディング ― 他人のコードから設計意図を読み取る技術

### ① なぜミドルに必要か

実務のコード作業は「書く2割、読む8割」。障害調査も機能追加もレビューも、まず既存コードを読むところから始まる。そしてOSSを読めるかどうかで、ライブラリの挙動が不明なときに「ドキュメントに無いから分からない」で止まるか、「ソースを見たら3分で分かった」と進めるかが分かれる。エージェント開発なら、Anthropic SDKやLangGraphの内部を読めれば、リトライ挙動やストリーミングの実装を正確に把握でき、独自実装の設計判断にも根拠が持てる。読解力は書く力の上限も決める:良い設計を大量に読んだ人だけが良い設計を書ける。

### ② 本質的な概念

本質:**コードは上から順に読むものではなく、「問いを立てて、答えだけを探しに行く」もの**。小説ではなく辞書の読み方をする。

**初学者の誤解と正しい理解の対比:**

| 誤解 | 正しい理解 |
|------|-----------|
| 「1行目から順に全部読む」 | 目的の無い通読は9割忘れる。「この関数はエラー時に何を返すか?」のような具体的な問いを先に立てる |
| 「全部理解してから触る」 | 完全理解は不要。変更に必要な範囲+その境界(呼び出し元/先)だけ理解すれば触ってよい。境界の特定こそが読解 |
| 「読む=目で追う」 | 手を動かして読む。デバッガでステップ実行する、ログを仕込んで動かす、テストを書いて仮説を検証する。「実行して確かめられる仮説」に変換するのが読解 |
| 「変数名や処理を1つずつ追う」 | 先に構造を掴む:エントリポイント(処理の入口)→主要なデータの流れ→依存の方向。詳細はその後 |

**読解の型(OSSを読むときの手順):**

1. README・ドキュメントで「何をするものか」を10分で掴む
2. ディレクトリ構成とpublicなAPI(公開インターフェース)を眺め、地図を作る
3. 最小の使用例を動かし、デバッガで1回ステップ実行して主経路を体感する
4. 問いを1つ立てる(例:「リトライは何回?どの例外で?」)
5. 問いの答えに関係するファイルだけgrep・定義ジャンプで追う
6. 分かったことを図かメモに残す(残さないと翌週消える)

### ③ 言語横断の視点

**変わらない本質:**エントリポイントから追う、データの流れを掴む、依存の方向を見る、問い駆動で読む。これは言語どころかアセンブリでも同じ。

**言語固有の違い:**

| 観点 | C# | Python |
|------|-----|--------|
| 型情報 | 全箇所に型があるので、シグネチャだけで挙動をかなり推測できる | 型ヒントが無いコードもある。その場合はテストコードが型情報の代わりになる |
| 定義ジャンプ | Visual Studio / Rider でほぼ確実 | VS Code + Pylanceで可能だが、動的な仕組み(`getattr`、デコレータ)の先は追えないことがある |
| 実行時の中身確認 | デバッガ、`ToString()`オーバーライド | `breakpoint()`+`dir(obj)` / `vars(obj)`で実行時に構造を暴ける(動的言語の強み) |
| 読みにくさの源 | 過剰な抽象化(インターフェースの多段ラップ) | メタプログラミング(デコレータ、`__getattr__`)、モンキーパッチ |

Pythonは静的に追えない箇所が増える代わりに、実行時に何でも覗ける。**C#は「読んで分かる」、Pythonは「動かして分かる」に寄せる**、と補正すると読解速度が落ちない。

### ④ 設計例:読解ノートの実例(Anthropic Python SDKを読む)

```text
【問い】anthropic SDKはAPIエラー時に自動リトライするか?何回?どの条件で?

【手順】
1. GitHubで anthropics/anthropic-sdk-python を開く
2. "retry" でリポジトリ内検索 → _base_client.py がヒット
3. DEFAULT_MAX_RETRIES = 2 を発見(既定で2回リトライ)
4. _should_retry() を読む → 408/409/429/5xx とコネクションエラーが対象
5. リトライ間隔は指数バックオフ+ジッター(retry-afterヘッダがあれば優先)

【検証】
client = Anthropic(max_retries=0) で無効化できることをコードで確認
```

## C1. 運用と可観測性 ― 本番で何が起きたか後から分かる技術

### ① なぜミドルに必要か

開発環境では再現しない不具合が本番で起きる。そのとき手元にあるのはログだけ。ログ設計が悪いと「エラーが出ました」以上のことが分からず、A3のデバッグ手順の第一歩(観測)が踏み出せない。エージェント開発では特に切実で、「LLMがおかしな応答を返した」の調査には、そのとき送ったプロンプト全文・モデル名・トークン数・レイテンシ・リトライ回数が残っていないと手も足も出ない。可観測性(observability:システムの外部出力から内部状態を推測できる度合い)は、A3のデバッグ技術を本番環境に拡張したもの。

### ② 本質的な概念

本質:**ログは未来の自分への手紙。「その時点で分かっていた文脈」を添えて事実を残す**。

**初学者の誤解と正しい理解の対比:**

| 誤解 | 正しい理解 |
|------|-----------|
| 「printで文字列を出せばログ」 | 検索・集計できないログは本番では役に立たない。構造化ログ(key=value / JSON形式で出すログ)にして、後から「request_id=Xの全ログ」を1コマンドで抽出できる形にする |
| 「エラーの時だけログを出す」 | エラーログ単体では「そこに至る経緯」が消えている。正常系の要所(処理の開始/終了、外部呼び出し)にもINFOを置き、道筋を再構成できるようにする |
| 「ログレベルは適当」 | ERROR=対応が必要、WARN=今回は続行したが要注意、INFO=正常系の道しるべ、DEBUG=開発時のみ。基準を決めて守らないとERRORが狼少年化する |
| 「例外はcatchしてログに出せば安全」 | 握りつぶし+ログは「静かな失敗」を生む。原則は再スローかリッチな文脈付きの変換。catchして続行するのは続行が仕様として正しい場合のみ |

もう1つの柱が**相関ID**:1リクエスト(エージェントなら1タスク実行)ごとに一意なIDを発行し、全ログに付与する。分散した処理でも「この実行の全記録」を一本の線として追える。

### ③ 言語横断の視点

**変わらない本質:**構造化ログ、ログレベルの規律、相関ID、「メトリクス(数値の時系列)・ログ(イベントの記録)・トレース(処理の流れ)」という3分類。

**言語固有の違い:**

| 観点 | C# | Python |
|------|-----|--------|
| 標準的ライブラリ | `Microsoft.Extensions.Logging` + Serilog | 標準`logging` / 構造化なら`structlog` |
| 構造化の書き方 | メッセージテンプレート:`Log.Information("解析完了 {Count}件", n)` | `logger.info("analysis_done", count=n)`(structlog) |
| 文脈の引き回し | `ILogger`のスコープ / `Activity` | `contextvars` + structlogの`bind` |
| 業界標準 | OpenTelemetry(共通) | OpenTelemetry(共通) |

C#のメッセージテンプレート(`{Count}`が文字列連結ではなくプロパティとして記録される)を知っていれば、structlogのkey-value方式は同じ思想だとすぐ分かる。

### ④ コード例:エージェントのLLM呼び出しに文脈付きログを仕込む

**Python(structlog):**

```python

                 model="claude-sonnet-4-6",
                 latency_ms=int((time.perf_counter() - start) * 1000),
                 input_tokens=resp.usage.input_tokens,
                 output_tokens=resp.usage.output_tokens)
        return resp.content
    except Exception:
        log.exception("llm_call_failed",       # スタックトレース付きで記録し、
                      latency_ms=int((time.perf_counter() - start) * 1000))

```

## C2. セキュリティ基礎 ― 攻撃者の視点で自分のコードを見る技術

### ① なぜミドルに必要か

エージェントや社内ツールをWeb API化した瞬間、あなたのコードは攻撃対象になる。「社内だから大丈夫」は通用しない(社内ネットワーク侵入後の横展開は常套手段)。ミドルに求められるのは専門家レベルではなく、**典型的な脆弱性を作り込まない・レビューで気づける**水準。具体的には「このSQL、文字列連結してませんか」「このAPI、認可チェックどこですか」と言えること。さらにLLMエージェント特有の攻撃面としてプロンプトインジェクション(外部から取り込んだテキスト内の指示をLLMが実行してしまう問題)があり、ここは従来のWebセキュリティ知識の延長で考えられる。

### ② 本質的な概念

本質は1つ:**外部から来るデータはすべて攻撃コードかもしれない、という前提でコードを書く**。入力の検証、出力のエスケープ、権限の確認。全脆弱性対策はこの3つの適用形にすぎない。

**初学者の誤解と正しい理解の対比:**

| 誤解 | 正しい理解 |
|------|-----------|
| 「SQLインジェクション対策=入力から危険文字を除去」 | 除去(サニタイズ)は漏れる。正解はプレースホルダ(パラメータ化クエリ):SQLとデータを構造的に分離する。EF CoreやSQLAlchemyを普通に使えば自動的にこれになる。危ないのは生SQLの文字列連結 |
| 「認証していれば安全」 | 認証(誰か)と認可(何をしていいか)は別物。ログイン済みユーザーがURLのIDを書き換えて他人のデータを見られる問題(IDOR)は認可の欠落 |
| 「パスワードは暗号化して保存」 | 暗号化(復号できる)ではなくハッシュ化(復号できない)。しかも専用アルゴリズム(bcrypt/argon2)を使う。SHA-256単発は高速すぎて総当たりに弱い |
| 「HTTPSにすれば安全」 | HTTPSは通信経路の保護のみ。アプリ内部の脆弱性(インジェクション、認可漏れ)には無力 |

### ③ 言語横断の視点

**変わらない本質:**OWASP Top 10(Webアプリの代表的脆弱性トップ10リスト)の全項目、「信頼境界」の考え方、認証と認可の区別。脆弱性はプロトコルとアーキテクチャの問題なので、ほぼ完全に言語非依存。

**言語固有の違い(道具の名前だけ):**

| 観点 | C# (ASP.NET Core) | Python (FastAPI) |
|------|-------------------|------------------|
| パラメータ化クエリ | EF Core標準 / `SqlParameter` | SQLAlchemy標準 / プレースホルダ |
| 入力検証 | データアノテーション / FluentValidation | Pydantic(型+制約で自動検証) |
| 認証基盤 | ASP.NET Core Identity / JWT Bearer | `fastapi.security` + python-jose等 |
| 依存パッケージ検査 | `dotnet list package --vulnerable` | `pip-audit` |

JWT認証をC#で組んだ経験があるなら、FastAPIでの実装は「同じ設計図の別言語版」でしかない。

### ④ コード例:SQLインジェクションの発生と対策

```python
# NG: 文字列連結。vin に「' OR '1'='1」が来たら全件漏洩

```

```csharp

// OK: EF Coreの補間文字列版は内部でパラメータ化される
```

## C3. パフォーマンス計測 ― 「遅い」を数字で語る技術

### ① なぜミドルに必要か

「なんか遅いので速くしておいて」という依頼に対し、ジュニアは怪しそうなコードを勘で書き換える。ミドルはまず計測し、「処理時間の82%がこの1関数です。ここを直せば5秒→1秒、他を直しても誤差です」と数字で語る。推測で行った最適化は、効果ゼロどころか可読性を犠牲にした分だけマイナスになることが多い。「**推測するな、計測せよ**」はこの分野の唯一にして絶対のルール。逆に言えば、計測の道具の使い方さえ知っていれば、ジュニアでもミドルの仕事ができる領域でもある。

### ② 本質的な概念

本質:**最適化は「計測→ボトルネック特定→修正→再計測」のループであり、計測で始まり計測で終わる**。

**初学者の誤解と正しい理解の対比:**

| 誤解 | 正しい理解 |
|------|-----------|
| 「経験者はコードを見れば遅い場所が分かる」 | 熟練者ほどプロファイラ(実行時間の内訳を計測するツール)を使う。人間の勘は外れるというのが業界の共通見解 |
| 「速いコード=良いコード。常に最適化すべき」 | 大半のコードは速度が問題にならない。要件(例:応答2秒以内)を満たしていれば最適化しない。読みやすさとのトレードオフを常に意識する |
| 「1回測って比較すればいい」 | 実行時間はゆらぐ(キャッシュ、JIT、GC、OSのスケジューリング)。複数回実行して分布(中央値・パーセンタイル)で比較する。ベンチマークFWはこれを自動でやってくれる |
| 「アルゴリズムを変えるのが最適化」 | 実務のボトルネックの多くはI/O(N+1クエリ、直列のAPI呼び出し、無駄なファイル読み込み)。B1・B2の知識がそのまま最適化手段になる |

計測は2階建て:(1)**プロファイリング**で「どこが遅いか」を特定し、(2)**マイクロベンチマーク**で「修正前後の差」を統計的に比較する。順番を逆にしない(いきなりベンチマークを書いても、そこがボトルネックでなければ無意味)。

### ③ 言語横断の視点

**変わらない本質:**計測→特定→修正→再計測のループ、計算量(O記法)の感覚、「I/Oは計算より桁違いに遅い」というレイテンシ感覚(メモリアクセス:ns、SSD:μs、ネットワーク:ms)。

**言語固有の違い:**

| 観点 | C# | Python |
|------|-----|--------|
| プロファイラ | Visual Studio Profiler / dotTrace | `cProfile`(標準)+ snakeviz(可視化)/ py-spy(本番プロセスに後付け可能) |
| マイクロベンチマーク | BenchmarkDotNet(統計処理込みの決定版) | `timeit`(標準)/ pytest-benchmark |
| 言語特有の速度事情 | JITコンパイルで概ね高速。GC圧(大量の短命オブジェクト)が典型的ボトルネック | インタプリタなのでピュアPythonのループは遅い。数値計算はNumPy等C実装のライブラリに寄せるのが定石 |

Pythonでは「ループを自分で書かずライブラリに任せる」が最適化の第一手になる点がC#との大きな文化差。

### ④ コード例:プロファイル→特定→修正

```python
# 1. プロファイルで「どこが遅いか」を特定
#    python -m cProfile -s cumulative analyze.py の出力(抜粋):
#      ncalls  cumtime  function
#       10000    8.21   parse_line      ← 全体9.0秒のうち8.2秒がここ
#       10000    7.90     re.compile    ← 犯人: ループ内で毎回正規表現をコンパイル

# NG: ループのたびにコンパイル
import re
def parse_line(line: str) -> str | None:
    m = re.compile(r"(?P<code>[PBCU]\d{4}),(?P<sev>\d+)").match(line)  # 毎回コンパイル
    return m and m.group("code")

# OK: コンパイルを1回に(モジュールレベルへ移動)
_DTC_RE = re.compile(r"(?P<code>[PBCU]\d{4}),(?P<sev>\d+)")
def parse_line(line: str) -> str | None:
    m = _DTC_RE.match(line)
    return m and m.group("code")

# 2. 修正前後を timeit で統計比較
#    python -m timeit -s "from analyze import parse_line" "parse_line('P0301,2')"
#    before: 8.4 usec per loop → after: 0.31 usec per loop(約27倍)
```

C#で同型の作業をするなら、Visual Studio ProfilerのCPU使用率ツールでホットパスを特定し、BenchmarkDotNetで`[Benchmark]`属性を付けたbefore/afterメソッドを比較する。**「プロファイラの出力を貼って、数字で改善を報告する」**という成果物の形は両言語で同じ。

### ⑤ 確認問題

1. 自分のDTC解析ツール(またはエージェント)をcProfile/VSプロファイラにかけ、累積時間トップ3の関数を特定してレポートせよ。
2. 「リストへの`in`検索(O(n))をsetに変える(O(1))」改善が効果を持つのは、要素数と検索回数がどんな条件のときか。timeitで境界を実測せよ。
3. 「LLM呼び出しを含むエージェントの応答が遅い」場合、プロファイラより先に確認すべき計測値は何か(ヒント:C1のログ設計)。理由とともに答えよ。

| 1 | Python公式プロファイラドキュメント https://docs.python.org/ja/3/library/profile.html | cProfileの節を読み、その場で自分のコードにかける |
| 2 | 『ハイパフォーマンスPython 第2版』(オライリー) | 1〜2章(計測)と6章(NumPy)のみ。全部は読まない |

理由:(1) WPFデスクトップ開発の現場はテスト文化が薄いことが多く、経験の穴になりやすい。(2) DI/インターフェースの経験が既にあるので、テスタビリティの話が最短で腹落ちする(前提知識が揃っている)。(3) Pythonのpytestは学習コストが低く、C#の知識をPythonに輸出する最初の成功体験に最適。(4) ISO 26262環境ではテスト成果物の言語化がそのまま業務評価につながる。

**2位:A1 設計原則(「経験」を「知識」に変換する)**
理由:MVVM+DIを半年使った今のあなたは「体で知っているが言語化できていない」状態。cosmicpythonでPython版のRepository/DIパターンを写経すると、「C#で毎日やっていたアレはこういう原理だったのか」という接続が起きる。これが「どの言語でも同じ原則で考えられる」状態の入口。エージェント開発のコード(Anthropic API直呼び)は、LLMクライアントをProtocolで抽象化する題材として完璧。

**3位:A3 デバッグ戦略(ドメインとの相乗効果)**
理由:FMEA/DRBFMの「故障モードから系統的に原因を遡る」思考を既に業務で使っている。デバッグシートの型はその応用なので習得が速く、かつ「車載の品質手法をソフトウェアデバッグに応用できる」という他のエンジニアにない強みになる。

**4位:B1 非同期(エージェント開発の実需)**
理由:C#のasync/awaitを使えている今、Pythonのasyncioとの実行モデル差(スレッドプールvsイベントループ、GIL)を学ぶと理解が立体になる。エージェントの並行LLM呼び出し・レート制限対応という実需が目の前にある。

**5位:B2 DB / 6位:B3 読解 / 7位以降:C群**
B2はEF Core経験があるためSQLAlchemyの学習コストが低い(Unit of Work思想が共通)。B3は毎週の習慣として並走させる(集中学習より反復が効く)。C群は「型だけ先に入れて、必要時に深掘り」で十分。ただしC1(構造化ログ)だけはエージェント開発で即実益が出るので、月次計画では前倒しで組み込む。
